Tài liệu này được biên soạn để giúp Giảng viên nhanh chóng đánh giá và trả lời các câu hỏi phản biện cốt lõi về chất lượng của đồ án HuTaBoo. Mọi nhận định kỹ thuật đều được gắn kèm đường dẫn file nguồn, tên lớp, tên hàm, và dòng code thực tế trong hệ thống.

CÂU 1: ỨNG DỤNG CÓ GIẢI QUYẾT ĐÚNG VẤN ĐỀ ĐÃ CHỌN KHÔNG?
--------------------------------------------------------------------------------
TRẢ LỜI: CÓ. 
Dự án giải quyết trọn vẹn bài toán mạng xã hội chia sẻ sách và đánh giá sách của người dùng (HuTaBoo).

MINH CHỨNG NGHIỆP VỤ & MÃ NGUỒN:
1. Đăng ký & Đăng nhập bảo mật:
   - Giao diện: Trang đăng nhập/đăng ký tích hợp tại `public/discover.html` và `public/reviews.html` thông qua các modal pop-up.
   - API Backend: Endpoint `/api/auth/register` (routes/review.routes.js: dòng 15) và `/api/auth/login` (routes/review.routes.js: dòng 16) kết hợp thư viện bcryptjs để băm mật khẩu và jsonwebtoken để cấp token phiên.
2. Quản lý tác phẩm (Sách):
   - Người dùng xem danh sách sách mẫu, tìm kiếm sách theo từ khóa (tiêu đề, tác giả, tag) và xem mô tả chi tiết của sách.
   - Giao diện: Slider và danh sách sách tại trang khám phá `public/discover.html` kết hợp file JS xử lý `public/js/discover.js`.
   - API Backend: `/api/books` (routes/review.routes.js: dòng 27) gọi đến phương thức `BookRepository.getAll()` và `BookRepository.searchBooks(keyword)` (repositories/review.repository.js: dòng 38, 138) để truy vấn MySQL.
3. Viết và quản lý đánh giá sách (Review & Rating):
   - Người dùng đăng nhập để viết review cho một tác phẩm (đánh giá rating từ 1 đến 5 sao, nội dung dài tối thiểu 5 ký tự) và đính kèm hình ảnh minh họa (ảnh upload).
   - Giao diện: Modal viết review tại `public/reviews.html` thông qua file JS xử lý `public/js/reviews.js`.
   - API Backend: `/api/reviews` (routes/review.routes.js: dòng 37) gọi đến `ReviewController.create()` (controllers/review.controller.js: dòng 174).
4. Viết và quản lý bình luận tương tác (Comment):
   - Người dùng viết bình luận dưới các bài review để trao đổi thảo luận.
   - API Backend: `/api/reviews/:reviewId/comments` (routes/review.routes.js: dòng 45-47) tương tác qua `CommentController` (controllers/review.controller.js: dòng 336).
5. Tự động cập nhật ảnh bìa động:
   - Khi viết review cho một sách tự thêm mới chưa có ảnh bìa cố định, hệ thống tự động copy ảnh đầu tiên từ bài review sớm nhất làm ảnh bìa của sách.
   - Mã nguồn backend xử lý: `BookRepository.updateCoverFromReview(bookId)` (repositories/review.repository.js: dòng 171).

2. Mở terminal tại thư mục gốc dự án và chạy:
   ```bash
   npm install
   npm run dev
   ```
3. Ứng dụng Desktop Electron sẽ tự động khởi chạy và hiển thị giao diện.

  - Tự động tạo cơ sở dữ liệu `hutaboo` nếu chưa tồn tại (dòng 34): `CREATE DATABASE IF NOT EXISTS \`hutaboo\``

CÂU 3: KIẾN TRÚC, MODULE, LỚP CÓ ĐƯỢC THIẾT KẾ CÓ CHỦ ĐÍCH KHÔNG?
--------------------------------------------------------------------------------
TRẢ LỜI: CÓ. 
Dự án được thiết kế có chủ đích rõ ràng theo kiến trúc phân lớp Clean/Layered Architecture, đảm bảo tính tách biệt trách nhiệm (Separation of Concerns).

CẤU TRÚC PHÂN LỚP THỰC TẾ TRONG MÃ NGUỒN:
1. Presentation Layer (Tầng hiển thị - Frontend):
   - Đặt tại thư mục `public/`.
   - Chứa giao diện tĩnh `public/discover.html` và `public/reviews.html`.
   - Các file script trong `public/js/` điều phối luồng giao diện và gọi fetch API. Nghiệp vụ frontend được tách thành các hook như `public/js/useBooks.js` (xử lý tải sách, tìm kiếm) và `public/js/useReviews.js` (xử lý đăng/xóa review, bình luận).
2. Routing Layer (Tầng định tuyến API):
   - Đặt tại file [routes/review.routes.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/routes/review.routes.js).
   - Tiếp nhận các HTTP request, gắn middleware kiểm tra token bảo mật `authenticateToken` và điều phối sang các Controller tương ứng.
3. Controller Layer (Tầng điều phối nghiệp vụ):
   - Đặt tại file [controllers/review.controller.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/controllers/review.controller.js).
   - Chứa các controller nghiệp vụ: `AuthController`, `BookController`, `ReviewController`, `CommentController`, `CategoryController`.
   - Nhiệm vụ: Tiếp nhận dữ liệu từ req.body/params, điều phối nghiệp vụ, xác thực JWT, phân quyền tài khoản (ví dụ chỉ tác giả mới được sửa/xóa bài viết của mình - dòng 225, 268) và trả về response HTTP.
4. Domain Model Layer (Tầng nghiệp vụ & thực thể cốt lõi):
   - Đặt tại file [models/review.model.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/models/review.model.js).
   - Chứa các lớp thực thể: `User`, `Book`, `Review`, `Comment`.
   - Nhiệm vụ: Định nghĩa cấu trúc đối tượng nghiệp vụ và tự đóng gói logic kiểm tra tính hợp lệ (validation) đầu vào (ví dụ: `Review.validate` dòng 52, `Comment.validate` dòng 81).
5. Data Access Layer / Repository Layer (Tầng tương tác cơ sở dữ liệu):
   - Đặt tại file [repositories/review.repository.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/repositories/review.repository.js).
   - Chứa các Repository: `UserRepository`, `BookRepository`, `ReviewRepository`, `CommentRepository`, `CategoryRepository`.
   - Nhiệm vụ: Thực hiện các câu lệnh truy vấn SQL trực tiếp qua connection pool của MySQL. Giúp che giấu chi tiết kỹ thuật lưu trữ khỏi tầng Controller.

CÂU 4: DESIGN PATTERN ĐƯỢC DÙNG Ở ĐÂU, VÌ SAO CẦN DÙNG?
--------------------------------------------------------------------------------
TRẢ LỜI: Dự án áp dụng có chủ đích 3 mẫu thiết kế (Design Patterns) kinh điển:

1. SINGLETON PATTERN:
   - Nơi sử dụng: Lớp `Database` tại [db/database.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/db/database.js).
   - Đoạn mã cụ thể (dòng 4-7, 23, 381-383):
     ```javascript
     class Database {
       constructor() {
         if (Database.instance) {
           return Database.instance;
         }
         ...
         Database.instance = this;
       }
     }
     const dbInstance = new Database();
     module.exports = dbInstance;
     ```
   - Vì sao cần dùng: Đảm bảo toàn bộ ứng dụng chỉ khởi tạo một connection pool duy nhất kết nối tới MySQL toàn cục. Việc này ngăn chặn tình trạng rò rỉ kết nối (connection leaks), giảm thiểu quá tải CPU/RAM của hệ thống cơ sở dữ liệu khi ứng dụng mở rộng.

2. REPOSITORY PATTERN:
   - Nơi sử dụng: Triển khai tại [repositories/review.repository.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/repositories/review.repository.js) và export các instance đơn lẻ đại diện (dòng 602-610):
     ```javascript
     module.exports = {
       UserRepository: new UserRepository(),
       BookRepository: new BookRepository(),
       ReviewRepository: new ReviewRepository(),
       CommentRepository: new CommentRepository(),
       CategoryRepository: new CategoryRepository()
     };
     ```
   - Vì sao cần dùng: Tách rời hoàn toàn tầng xử lý HTTP/Nghiệp vụ (Controller) khỏi tầng truy vấn vật lý (SQL Query). Nếu sau này dự án chuyển đổi từ MySQL sang cơ sở dữ liệu khác (như PostgreSQL, MongoDB), ta chỉ cần viết lại các hàm SQL trong Repository mà không cần thay đổi bất kỳ logic nào của Controller hay Router.

3. OBSERVER PATTERN (PUBLISH-SUBSCRIBE):
   - Nơi sử dụng: Triển khai ở Client/Frontend tại các hook `public/js/useAuth.js` và `public/js/useReviews.js`.
   - Ý nghĩa: Các module này cung cấp hàm đăng ký sự kiện lắng nghe (subscribe). Khi trạng thái đăng nhập thay đổi hoặc danh sách bài review mới được thêm/sửa/xóa, các listener đã đăng ký trên giao diện sẽ tự động được gọi (notify) để render lại UI tức thời mà không cần reload trang web.

CÂU 5: SOLID ĐƯỢC THỂ HIỆN CỤ THỂ TRONG CẤU TRÚC LỚP/MODULE NÀO?
--------------------------------------------------------------------------------
TRẢ LỜI: Thiết kế mã nguồn của dự án áp dụng chặt chẽ các nguyên lý SOLID:

1. SINGLE RESPONSIBILITY PRINCIPLE (SRP - Nguyên lý đơn nhiệm):
   - Thể hiện: Mỗi lớp/file chỉ giữ một trách nhiệm duy nhất.
   - [models/review.model.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/models/review.model.js) chỉ định nghĩa kiểu thực thể và thực hiện kiểm tra tính hợp lệ (validate).
   - [repositories/review.repository.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/repositories/review.repository.js) chỉ đảm nhận việc lập câu lệnh SQL và lấy dữ liệu thô.
   - [controllers/review.controller.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/controllers/review.controller.js) chỉ chịu trách nhiệm nhận request, trả response HTTP và điều hướng luồng.

2. OPEN/CLOSED PRINCIPLE (OCP - Nguyên lý đóng/mở):
   - Thể hiện: Thiết kế mở rộng nhưng đóng đối với việc chỉnh sửa.
   - Khi người dùng viết review, họ có thể thêm các thể loại sách tùy chỉnh mới (custom categories). Hệ thống sẽ tự động chèn vào bảng `categories` và tạo liên kết thông qua bảng trung gian `review_categories` (được định nghĩa ở db/database.js dòng 123-131 và repositories/review.repository.js dòng 424-444). Cơ chế này cho phép mở rộng vô hạn các thể loại sách mới mà không cần sửa đổi cấu trúc cốt lõi của bảng `reviews` hay sửa đổi logic của Model `Review`.

3. LISKOV SUBSTITUTION PRINCIPLE (LSP - Nguyên lý thay thế Liskov):
   - Thể hiện: Các đối tượng thuộc lớp `Book`, `Review`, `Comment` đều đảm bảo giữ đúng giao diện thuộc tính định sẵn (mã nguồn models/review.model.js) và có thể được truyền thay thế linh hoạt vào bất kỳ hàm xử lý định dạng hoặc hiển thị dữ liệu nào của hệ thống mà không làm crash ứng dụng.

4. INTERFACE SEGREGATION PRINCIPLE (ISP - Nguyên lý phân tách Interface):
   - Thể hiện: Thay vì gộp chung toàn bộ phương thức truy vấn cơ sở dữ liệu vào một class database dùng chung duy nhất, dự án chia nhỏ các phương thức thành các Repository chuyên biệt riêng lẻ: `UserRepository` (chỉ xử lý người dùng), `BookRepository` (chỉ xử lý sách), `ReviewRepository` (chỉ xử lý bài review), `CommentRepository` (chỉ xử lý bình luận) tại repositories/review.repository.js.

5. DEPENDENCY INVERSION PRINCIPLE (DIP - Nguyên lý đảo ngược phụ thuộc):
   - Thể hiện: Tầng nghiệp vụ cấp cao (Controller) không phụ thuộc trực tiếp vào database MySQL vật lý, mà phụ thuộc vào tầng trừu tượng (các Repository). 
   - Minh chứng rõ nhất trong kiểm thử: Tại file unit test [controllers/review.controller_test.js](file:///d:/NGUYENHUONG/HOC/PTUD/BTL/controllers/review.controller_test.js), ta có thể mock hoàn toàn các Repository một cách dễ dàng và tiêm (inject) vào Controller để kiểm thử logic xử lý mà không cần khởi động kết nối cơ sở dữ liệu MySQL thật.
